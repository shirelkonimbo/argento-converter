# PLAN — מיזוג אקסלים של ספק Argento לאקסל וריאנטים מאוחד

## מטרה
עמוד React עצמאי (`converter.html`, נפתח ישירות בדפדפן, בלי שרת/build) שמאפשר להעלות את קבצי ה-`.xls`/`.xlsx` שהספק שולח, מפרק ומתרגם את התוכן בצד הלקוח, ומאפשר להוריד **קובץ xlsx מאוחד אחד** — שורה אחת לכל SKU/וריאנט.

שיראל תשתמש בעמוד הזה בכל פעם שמגיעים קבצים חדשים מהספק: מעלה קבצים → לוחצת "הרץ המרה" → מורידה את הקובץ המוכן.

---

## ממצאים ממבנה קבצי המקור (נבדק בפועל)

כל קובץ מכיל 2 sheets: `Regular Tag` ו-`New Tag`.
עמודות (שורה 2 בכל sheet היא הכותרת, הנתונים מתחילים בשורה 3):

| עמודה | תוכן |
|---|---|
| No. | מספר פנימי של הספק |
| ARG code | קוד המוצר, למשל `R1000509-1-1-7`, לפעמים עם `*` בהתחלה |
| Photo | תמיד ריק |
| Description | טקסט חופשי: סוג בסיס (`earring`/`necklace`/...) ולעיתים שורה שנייה עם תת-סוג/צורה/מידה |
| quantity | כמות |
| Size | מידת טבעת בפורמט `#7#` או ריק |
| tag price | מחיר תג |
| Plating color | טקסט חופשי: `rhd` / `14K` / `Rose` / `Brass` / תיאור אבן משולב |
| Remark | הערה חופשית, לרוב `PI:XXXX Metal-plating` |

### מבנה קוד ARG (מאומת מול הנתונים)
```
[*]LETTER + BASE_NUMBER + "-" + PLATING_DIGIT + "-" + STONE_DIGIT + "-" + SIZE
```
- `*` בתחילת הקוד = תג "חדש" (ולא ה-sheet name — יש קבצים בהם `*` הוא הסימן היחיד האמין).
- `LETTER`: E=earring, N=necklace, B=bracelet/hand bracelet/bangle, R=ring
- `PLATING_DIGIT`: 1=rh, 2=gold, 3=rose — מאומת מול עמודת Plating color (`rhd`↔1, `14K`↔2, `Rose`↔3)
- `STONE_DIGIT`: 0=plain, 1=white, 2=blue, 3=green, 4=black, 6=red, 7=aqua, 8=pink
  - נמצאו גם ערכים לא-מתועדים (למשל `9`) — **מוצג כמו שהוא + מדגל לבדיקה ידנית** (לא מפרש/מנחש)
- `SIZE`: מספר (מידת טבעת/צמיד) או `A` אם אין מידה

### שורות פגומות שנמצאו
כ-19 שורות ריקות/חלקיות + כמה קודים עם מקף כפול (ספרת פלטינג חסרה), למשל:
`N1000509--1-1-A`
**החלטה: שורות כאלה מדולגות מהפלט הראשי, ונרשמות בעמודות שגיאה + נכללות בגיליון "שגיאות" נפרד בקובץ הפלט.**

### דוגמאות לתוכן עמודת Description (מציג את המורכבות)
```
earring
earring\n9mm
earring\n5*19mm        (מידה דו-ממדית)
stud earring           (תת-סוג בלי מידה)
necklace\nmarquise     (צורה, לא מידה)
ring\ndrop
choker\n4*8mm
bracelet\n1.75mm
long necklace
ear cuff
earring\n耳圈12mm | 心形8*7mm   (טקסט בסינית — עמודת raw נשארת כמו שהיא)
```

## החלטות שהתקבלו בשיחה

1. **קוד ARG לא נבנה מחדש — הוא מפורק** לעמודות נפרדות (letter/base/plating/stone/size), כי הוא כבר מגיע בפורמט שדומה למה שנדרש.
2. **סטטוס "חדש"** נקבע לפי `*` בתחילת קוד ה-ARG (לא לפי שם ה-sheet).
3. **ספרת אבן לא מוכרת** (לדוגמה `9`): מוצגת כמו שהיא + עמודת דגל "לבדיקה", לא מנוחשת.
4. **קוד פגום** (מקף כפול / ספרה חסרה): השורה מדולגת מהפלט הראשי + נרשמת בגיליון שגיאות נפרד.
5. **פירוק Description** ל-3 עמודות: סוג-בסיס (base type) / תת-סוג (sub type, כשקיים) / מידה-גולמית (raw size/shape text, ללא פענוח).
6. **גרנולריות פלט**: שורה אחת לכל SKU (כל וריאנט = שורה נפרדת, לא מקובץ).
7. **היררכיית קטגוריה**: קטגוריה (מהאות) → תת-קטגוריה (מה-Description) → צבע (מפירוק plating+stone).
8. **תרגום לעברית**: המשתמש (שיראל) ישלים מילון תרגום מלא בעצמו — הסקריפט קורא מילון חיצוני ניתן לעריכה (קובץ JSON), ואם מונח חסר במילון — משאיר את המקור באנגלית + מדגל "אין תרגום".
9. **עמודות מקור נשמרות** — אבל בגיליון נפרד ("מקור"), לא בתוך גיליון ה-Variants הראשי (כדי לא להעמיס אותו במידע טכני): שם קובץ מקור, מספר CI, שם sheet, מספר שורה, No. של הספק, קוד ARG (לצורך התאמה בין הגיליונות).
10. **מימוש**: עמוד React יחיד (`converter.html`) שרץ כולו בדפדפן — לא סקריפט Node ולא נוסחאות אקסל. קריאת xls/xlsx בעזרת SheetJS (`xlsx`), כתיבת הפלט בעזרת ExcelJS (כדי לתמוך ב-dropdown/data validation אמיתי בעמודות קטגוריה/תת-קטגוריה/אבן — SheetJS בגרסה החופשית לא כותב dropdown).
11. **גיליון Variants מנוקה מכפילויות**: אין בו יותר ספרות/אותיות גולמיות (type_letter, plating_digit, stone_digit) ואין עמודות EN כפולות לצד עמודות עבריות מתורגמות — רק התוצאה המתורגמת/סופית. כל הדגלים (אבן לא מוכרת / אין תרגום / אין התאמת צורה) מתקצרים לעמודת "לבדיקה" אחת, עם הפירוט המלא בגיליון Flags.
12. **מידה משולבת**: אם לסיפית הקוד יש ערך מספרי (למשל מידת טבעת/צמיד) — הוא מוצג; אם הסיפית היא `A` אבל בטקסט ה-Description יש מידה (מכיל ספרה, למשל "9mm"/"5*19mm") — הטקסט הזה מוצג במקום. מילות צורה (heart/round/marquise...) לא נחשבות "מידה" (אין בהן ספרה) ועוברות לניסיון התאמת צורה.

## היררכיית קטגוריות החנות (התקבלה מהמשתמש, לצורך רפרנס/עתיד)

עץ קטגוריות אמיתי בחנות — 4 קטגוריות עליון, כל אחת עם רמת חומר/ציפוי, ואז רשימת תת-סוגים (חלקם עם תת-תת-סוג):

**כל הטבעות**
- חומר: טבעות כסף / טבעות ציפוי זהב / טבעות רוזגולד
- תת-סוג: טבעות קלאסיות, טבעת זרת, טבעת חותם, טבעת שורה, טבעת לאירוע, טבעת ללא שיבוץ

**כל הצמידים**
- חומר: צמידי כסף / צמידי ציפוי זהב / צמידי רוזגולד
- תת-סוג: צמיד קשיח-בנגל (תת-תת: שורה / אלמנט / ללא שיבוץ), צמיד לרגל, צמיד טניס (תת-תת: שיבוץ לבן / שיבוץ שחור), צמיד עדין, צמיד אלמנט, צמיד לאירוע, צמיד חוליות

**כל השרשראות**
- חומר: שרשראות כסף / שרשראות ציפוי זהב / שרשראות רוזגולד
- תת-סוג: שרשרת עניבה, שרשרת טניס (מידות), שרשרת צ'וקר, שרשרת תליון, שרשרת קשיחה, שרשרת אלמנטים

**כל העגילים**
- חומר: עגילי כסף / עגילי ציפוי זהב / עגילי רוזגולד
- תת-סוג: עגיל צמוד, עגיל ארוך, עגיל 2 חורים, עגיל תלוי, חישוקים (מידות: XL/L/M/S/XS/Y), עגיל מטפס, פירסינג (תת-תת: הליקס / טרגוס), תוספות לעגילים (תת-תת: סוגר עם אלמנט / סוגר עם שרשרת)

**רשימת פילטרים נלווית (facets):** קטגוריה, צבע, חומר, מידה, צורה-תליון, אבן, מחיר.
**מילון "צורות" (shape):** סוליטר, פרפר, לב, לוטוס, מגן דוד, חמסה, דגים, אותיות, חי, עין, טיפה, עגולה, מרקיזה, אובלית, מלבן.

### החלטה — איך זה מתחבר לפלט
- **קטגוריה נגזרת אוטומטית** מ-`type_letter` (E/N/B/R). שם הפלטינג נגזר מ-`plating_digit` — שתיהן עמודות **dropdown** בפלט (רשימת האפשרויות מגיעה מגיליון עזר חבוי `Lists`), כדי שאפשר לתקן ידנית בקליק אם צריך.
- **תת-הסוג (subcategory) לא נקבע אוטומטית** — אין בקובצי הספק מידע מספיק להבחין בין למשל "טבעת חותם" ל"טבעת שורה" או "עגיל צמוד" ל"חישוקים". עמודת `תת-קטגוריה` בפלט היא **dropdown ריק** עם רשימה אחת שטוחה של כל 34 תתי-הקטגוריות מכל 4 העצים (לא מסונן לפי קטגוריה נבחרת — Excel data validation דינמי-לפי-עמודה-אחרת מורכב הרבה יותר ולא הוחלט שנדרש) — שיראל בוחרת מהרשימה, לא כותבת חופשי.
- **עמודת "צורה" (עברית)** — הסקריפט מנסה להתאים מילה ממילון ה-shapes (round/marquise/heart/drop/oval/rectangle/butterfly/star-of-david/hamsa/fish/eye/solitaire...) על טקסט השורה השנייה ב-Description, **רק אם הטקסט הזה לא מכיל ספרה** (טקסט עם ספרה מטופל כמידה, לא כצורה). אם לא נמצאה התאמה — העמודה נשארת ריקה והשורה מסומנת "לבדיקה".
- **עמודת "שם אבן"** גם היא dropdown (8 האפשרויות הידועות), כדי לאפשר תיקון ידני מהיר אם הספרה שהתקבלה מהקוד לא ברשימה הידועה.

עודכן ל-4 רמות היררכיה + הפרדה בין צורת אבן לצורת תכשיט + עמודת גודל אבן נפרדת ממידה:

| 3 | קטגוריה (רמה 1) | dropdown — כל הטבעות/כל הצמידים/כל השרשראות/כל העגילים (שם מדויק 1:1 לעץ בחנות) |

| 5 | צבע מתכת (רמה 2) | dropdown — כסף/זהב/רוזגולד (מתורגם מ-rh/gold/rose בקוד) |

| 8 | מידה | מסיפית קוד ה-ARG (מידת טבעת/צמיד מספרית), ריק אם `A` |
| 9 | סוג בסיס (עברית) | תרגום עברי; אם לא נמצא תרגום — מוצג הטקסט המקורי באנגלית + מסומן "לבדיקה" |
| 10 | צורת אבן | dropdown נפרד — עגולה/מרקיזה/אובלית/טיפה/לב/מלבן |
| 11 | צורה | dropdown נפרד (צורת תכשיט/תליון כללית) — סוליטר/פרפר/לב/לוטוס/מגן דוד/חמסה/דגים/אותיות/חי/עין/טיפה |

| 13 | תת-תת-קטגוריה (רמה 4) | dropdown ריק — הרמה העמוקה (שורה/אלמנט/שיבוץ לבן/שיבוץ שחור/הליקס/טרגוס/XL-XS וכו', ניתנת להוספה במילון) |
| 14 | כמות | quantity מהמקור |
| 15 | מחיר | tag price מהמקור |
| 16 | הערה | Remark (או Plating color החופשי אם Remark ריק) |
| 17 | לבדיקה | "כן" אם יש בעיה מכל סוג (אבן לא מוכרת / אין תרגום / אין התאמת צורה כלל) — פירוט מלא בגיליון Flags |

### Sheet 2: `מקור` — מידע טכני/מעקב, אותו סדר שורות כמו Variants
עמודות: קובץ מקור, מספר CI, Sheet מקור, שורה במקור, מספר ספק (No.), קוד ARG (למפתח התאמה בין הגיליונות).

עמודות: קובץ מקור, Sheet מקור, שורה במקור, קוד ARG, סיבה (למשל "קוד פגום — ספרה חסרה").

### Sheet 4: `Flags` — שורות תקינות שסומנו "לבדיקה", לצורך סקירה ממוקדת
עמודות: קוד ARG, קטגוריה, שם אבן, סוג בסיס, תת-סוג גולמי (מקור), סיבת דגל, קובץ מקור, שורה במקור.

### Sheet 5: `Lists` (חבוי) — רשימות המקור לכל ה-dropdowns (קטגוריה/תת-קטגוריה/אבן).

## קבצים שנבנו

```
argento/
  PLAN.md            (קובץ זה)
  converter.html     עמוד React עצמאי — נפתח ישירות בדפדפן, אין build/שרת/npm install
```

`converter.html` טוען מ-CDN: React, ReactDOM, Babel standalone (JSX בדפדפן), SheetJS (`xlsx`, לקריאת קבצי המקור) ו-ExcelJS (לכתיבת הפלט עם dropdowns). מילון התרגום (`base_types` + `shapes`) מוטבע בעמוד כ-state עם ערכי דוגמה, וניתן לעריכה ישירה בתיבת טקסט בממשק (JSON) לפני כל הרצה — אין קובץ חיצוני נפרד.

מילון דוגמה (state התחלתי בעמוד):
```json
{
  "base_types": {
    "earring": "עגיל", "stud earring": "עגיל צמיד", "ear cuff": "עגיל צמוד",
    "necklace": "שרשרת", "long necklace": "שרשרת ארוכה", "choker": "שרשרת צוואר",
    "bracelet": "צמיד", "hand bracelet": "צמיד יד", "bangle": "צמיד ריגלי", "ring": "טבעת"
  },
  "shapes": {
    "round": "עגולה", "marquise": "מרקיזה", "oval": "אובלית", "drop": "טיפה",
    "heart": "לב", "rectangle": "מלבן", "solitaire": "סוליטר", "butterfly": "פרפר",
    "star of david": "מגן דוד", "hamsa": "חמסה", "fish": "דגים", "eye": "עין"
  }
}
```
שיראל יכולה לערוך את התיבה הזו בממשק בכל הרצה (למשל להוסיף מונח חדש שהתגלה), התוכן לא נשמר אוטומטית בין הרצות (כל הרצה מתחילה מהערכים ההתחלתיים בקוד) — אם רוצים שהתוספות יישמרו לצמיד, יש לעדכן את ה-state ההתחלתי (`DEFAULT_TRANSLATIONS`) בקובץ עצמו.

## לוגיקת העמוד (`converter.html`) — שלבים

1. המשתמשת בוחרת קובץ/קבצי `.xls`/`.xlsx` בממשק (`<input type="file" multiple>`).
2. בלחיצה על "הרץ המרה": לכל קובץ נקרא `ArrayBuffer` ומפוענח עם `XLSX.read`, מעבר על שני ה-sheets (`Regular Tag`, `New Tag`).
3. לכל שורת דאטה (מדלגים על 2 שורות הכותרת):
   - דילוג על שורות ריקות לחלוטין, ודילוג שקט על שורות בלי קוד ARG כלל.
   - Regex על קוד ARG: `^(\*)?([A-Z]+)(\d+)-(\d+)-(\d+)-(.+)$` — אם לא תואם → נכתב ל-Errors עם הסיבה, לא ממשיך לפענוח.
   - מיפוי `type_letter` → קטגוריה, `plating_digit` → שם פלטינג, `stone_digit` → שם אבן (או משאיר את הספרה כמו שהיא אם לא ברשימה).
   - פיצול Description לפי שורות: שורה ראשונה = base type, שורה שנייה (אם קיימת) = subcategory_raw.
   - **מידה**: אם סיפית הקוד ≠ `A` → זו המידה. אחרת, אם ל-subcategory_raw יש ספרה בתוכו → הוא המידה.
   - **צורה**: רק אם ל-subcategory_raw אין ספרה בתוכו (כלומר לא טופל כמידה) → מחפשים התאמה במילון `shapes`.
   - חיפוש `base type` (lowercase) במילון `base_types`; אם לא נמצא → הצגת הטקסט המקורי + flag.
   - איסוף כל סיבות הדגל (אבן לא מוכרת / אין תרגום / אין התאמת צורה) לרשימה אחת → עמודת "לבדיקה" = "כן"/ריק, והפירוט נשמר לגיליון Flags.
4. בניית קובץ Excel עם ExcelJS: גיליון עזר חבוי `Lists` (רשימות ה-dropdown), `Variants`, `מקור`, `Errors`, `Flags`; הגדרת `dataValidation` מסוג list על עמודות קטגוריה/שם אבן/תת-קטגוריה בגיליון Variants, מפנה לטווחים בגיליון `Lists`.
5. `wb.xlsx.writeBuffer()` → `Blob` → קישור הורדה בממשק, עם סיכום מספרים (שורות תקינות / שגיאות / דגלים לבדיקה).

## מה עדיין פתוח / באחריות שיראל
- להשלים/להרחיב את מילון התרגום (`base_types` + `shapes`) בתיבת הטקסט בממשק בכל פעם שמתגלה מונח חדש (או לבקש עדכון קבוע ב-`DEFAULT_TRANSLATIONS` בקוד).
- **לבחור ידנית מה-dropdown את "תת-קטגוריה"** לכל שורה לפי עץ הקטגוריות המלא (טבעת חותם/עגיל צמוד/חישוקים/פירסינג וכו') — לא נקבע אוטומטית.
- לעבור על גיליון Flags ולתקן/להשלים שורות מסומנות "לבדיקה".
- לאשר את סדר/שמות העמודות הסופי בפלט לפני שילוב במערכת (Medusa/YouLeap) אם רלוונטי.
